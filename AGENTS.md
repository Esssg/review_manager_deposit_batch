## 1. 코딩 전에 생각하기

**가정하지 마세요. 혼란을 숨기지 마세요. 트레이드오프를 드러내세요.**

구현하기 전에:
- 가정을 명시적으로 밝히세요. 확실하지 않다면 질문하세요.
- 여러 해석이 가능하다면 조용히 하나를 고르지 말고 제시하세요.
- 더 단순한 접근이 있다면 말하세요. 필요하다면 이의를 제기하세요.
- 불명확한 점이 있다면 멈추세요. 무엇이 혼란스러운지 말하고 질문하세요.

## 2. 단순함 우선

**문제를 해결하는 최소한의 코드. 추측성 작업은 하지 않습니다.**

- 요청받은 것 이상의 기능을 추가하지 마세요.
- 한 번만 쓰는 코드를 위해 추상화를 만들지 마세요.
- 요청되지 않은 "유연성"이나 "설정 가능성"을 추가하지 마세요.
- 불가능한 시나리오를 위한 오류 처리를 추가하지 마세요.
- 200줄로 작성했지만 50줄로 가능하다면 다시 단순화하세요.

스스로에게 물어보세요: "시니어 엔지니어가 이걸 과하게 복잡하다고 볼까?" 그렇다면 단순화하세요.

## 3. 외과적인 변경

**반드시 필요한 부분만 건드리세요. 자신이 만든 어지러움만 정리하세요.**

기존 코드를 수정할 때:
- 주변 코드, 주석, 포맷을 "개선"하지 마세요.
- 고장 나지 않은 코드를 리팩터링하지 마세요.
- 본인이 다르게 작성하고 싶더라도 기존 스타일을 따르세요.
- 코드 주석은 비개발자도 이해할 수 있도록 쉬운 한국어로 작성하세요.
- 관련 없는 죽은 코드를 발견하면 언급만 하고 삭제하지 마세요.

변경으로 인해 불필요해진 항목이 생겼을 때:
- 본인의 변경 때문에 사용되지 않게 된 import, 변수, 함수는 제거하세요.
- 요청받지 않았다면 기존의 죽은 코드는 제거하지 마세요.

프로젝트 개요, 실행 방법, 환경변수, API 사용법이 바뀌면 `README.md` 변경안도 함께 업데이트하세요.

검증 기준: 변경된 모든 줄은 사용자의 요청과 직접 연결되어야 합니다.
